# Flyway + Hibernate Entity Validation

## When to Use

- Creating a new `@Entity` class
- Modifying any existing entity (`@Table`, `@Column`, `@Embedded`, `@Enumerated`, `@DiscriminatorColumn`)
- Adding `@Embedded AuditFields` to an entity
- Creating a subclass entity (extends Customer, Account, etc.)
- Running `./gradlew bootRun` with `ddl-auto: none`

## The Trap

`ddl-auto: create-drop` hides mismatches. Hibernate recreates the DB from entity metadata alone — Flyway never runs. Tests pass. But `ddl-auto: none` + `bootRun` fails because Flyway tables exist with different names.

**Rule: Tests passing is NOT proof that entity mappings are correct.**

## Validation Steps

### 1. Before writing entity code

Check `src/main/resources/db/migration/` for existing tables that cover your entity. If a Flyway migration already exists for the domain, read it first.

### 2. When writing entity code

For **every** `@Entity`, complete this checklist:

- [ ] `@Table(name = "...")` — explicit name matching Flyway `CREATE TABLE` exactly (check singular/plural)
- [ ] `@Column(name = "...")` — explicit name on every column, especially enum fields
- [ ] `@Embedded AuditFields` — Flyway migration MUST include `created_at, created_by, updated_at, updated_by`
- [ ] Field shadows `@DiscriminatorColumn`? Mark `@Column(insertable = false, updatable = false)`
- [ ] Subclass entity (`extends Customer`) — explicit `@Table` matching Flyway
- [ ] No `@Table` on entity → assume Hibernate defaults to singular class name, verify against Flyway

### 3. After entity is complete — verify with bootRun

```bash
./gradlew bootRun > /tmp/bootrun.log 2>&1 &
sleep 30
curl -s http://localhost:8080/api/products/search
# OR trigger the persist path relevant to your entity
grep -i "exception\|error" /tmp/bootrun.log
pkill -f bootRun
```

If the server starts and persists data without errors → entity is validated.

If `Column not found` or `Table not found` → fix entity annotations or add Flyway migration.

### 4. If you added a new table

Create a Flyway migration (`V{n}__descriptive_name.sql`) before committing. All tables must be in Flyway when `ddl-auto: none`.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| `extends Customer` without `@Table` | Add `@Table(name = "corporate_customers")` (check Flyway for exact name) |
| Enum field `type` shadows `@DiscriminatorColumn(name = "account_type")` | Add `@Column(name = "account_type", insertable = false, updatable = false)` |
| `AuditFields` embedded but migration missing audit columns | Add migration `ALTER TABLE ... ADD COLUMN created_at TIMESTAMP` etc. |
| `BIGSERIAL` in migration (PostgreSQL only) | Use `BIGINT GENERATED BY DEFAULT AS IDENTITY` for H2 compatibility |
| `CONSTRAINT...REFERENCES` inline FK (H2 unsupported) | Use `REFERENCES tab(id)` instead |

## Red Flags — STOP

- "Tests already pass, this is fine" → bootRun not run
- "I'll add the Flyway migration later" → Must be done before commit
- "The table name is obvious, I don't need `@Table`" → Hibernate default ≠ Flyway name
- "This is just a simple entity" → Simple entities cause the worst production failures

## Key Principle

**Every entity change requires two verifications:**
1. Tests pass (`./gradlew test`)
2. Server starts and persists (`./gradlew bootRun` + curl)

Skipping step 2 means you're shipping blind.
